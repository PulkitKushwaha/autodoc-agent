## System architecture

### Overview
AutoDoc is an agentic documentation generator that ingests Python codebases
and produces comprehensive, structured technical documentation through a
self-evaluating multi-agent pipeline. It accepts a GitHub URL or local path,
parses every Python file using AST analysis, builds a dependency graph,
infers the tech stack, coordinates five specialist LLM-powered writer agents,
passes all generated sections through a critic agent that scores quality and
flags weak sections for revision, then assembles and renders the final output
in Markdown, HTML, and PDF formats.

### Core components

**Ingestion engine** — accepts a GitHub URL or local path, parses every `.py`
file using Python's `ast` module, builds an internal import dependency graph,
and infers the tech stack from dependency files. Produces `CodebaseManifest`
— a fully validated Pydantic model saved to `output/manifest.json`.

**Agent pipeline** — a LangGraph `StateGraph` with `DocState` as shared state.
Planner seeds state, five writer agents produce sections, critic scores each
1-10 and flags weak ones for revision, revision router loops writers back with
critique context, assembler collects all sections into `final_docs`.

**Renderer layer** — takes `final_docs` dict and produces three output formats:
`documentation.md` (single assembled Markdown file), `site/` (multi-page static
HTML site), and `project_documentation.pdf` (PDF via WeasyPrint).

**CLI** — Typer app with `autodoc run --input --output --format` flags. Rich
progress bar shows live pipeline execution. Quality score table printed after
critic completes.

### Data flow

```
1.  autodoc run --input <path> --format all
2.  fetcher.py → local directory
3.  parser.py → list[FileInfo] via AST
4.  graph.py → DependencyGraph
5.  detector.py → StackInfo
6.  CodebaseManifest → manifest.json
7.  LangGraph pipeline invoked
8.  planner → architecture → api → db → auth → deploy → critic
9.  should_revise() → revision_router → writers (with critique) → critic
10. assembler → final_docs dict
11. render_markdown() → output/documentation.md
12. render_html_site() → output/site/index.html + section pages
13. render_pdf() → output/autodoc_documentation.pdf
```

### Key design decisions

**Iterative refinement** — critic scores each section independently, routes
only weak ones back. Writers receive specific feedback via `{{ critique }}`
Jinja2 variable on revision pass. Capped at `MAX_REVISIONS=2`.

**Strategy pattern for LLM** — `BaseLLMClient` ABC, one factory function,
one env var. `AUTODOC_USE_MOCK=false` + key → real Claude. Zero code changes.

**Jinja2 for both prompts and output** — same renderer pattern used for agent
prompts (`autodoc/prompts/`) and output templates (`autodoc/templates/`).

**WeasyPrint for PDF** — converts HTML + CSS to PDF natively in Python.
Print-specific CSS hides sidebar, sets A4 page size, adds page numbers.

## Data models

### Overview
AutoDoc uses Pydantic `BaseModel` subclasses for all ingestion output and
a `TypedDict` for LangGraph pipeline state. All data is in-memory per run
with `manifest.json` as the only persistence artifact.

### `CodebaseManifest` hierarchy

```

├── files: list[FileInfo]
```

## Authentication and security

### Overview
AutoDoc is a local CLI tool with no user-facing authentication. Security
is limited to API key management and safe repository handling.

### Security measures

| Measure | Implementation |
|---------|---------------|
| API key never hardcoded | Pydantic BaseSettings reads from `.env` only |
| API key never logged | No log statement references the key field |
| `.env` gitignored | Root `.gitignore` excludes `.env` |
| Temp repos cleaned up | `try/finally` in `cli.py` |
| Shallow clone only | `depth=1` in `git.Repo.clone_from()` |
| Mock mode default | `AUTODOC_USE_MOCK=true` — key never needed in dev |
| StrictUndefined Jinja2 | Prompt variable errors caught at dev time |
| JSON parsing hardened | Critic strips fences, returns empty on malformed |

### Switching to real LLM mode

```bash
# In .env
ANTHROPIC_API_KEY=sk-ant-your-key-here
AUTODOC_USE_MOCK=false
```

Zero code changes needed anywhere in the codebase.
