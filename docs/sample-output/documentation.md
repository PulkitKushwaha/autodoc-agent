## System architecture

### Overview
AutoDoc is an agentic documentation generator that ingests Python codebases
and produces comprehensive, structured technical documentation through a
self-evaluating multi-agent pipeline. It accepts a GitHub URL or local path,
parses every Python file using AST analysis, builds a dependency graph,
infers the tech stack, coordinates five specialist LLM-powered writer agents,
passes all generated sections through a critic agent that scores quality and
flags weak sections for revision, then assembles and renders the final output
in Markdown, HTML, and PDF formats.

### Core components

**Ingestion engine** вЂ” accepts a GitHub URL or local path, parses every `.py`
file using Python's `ast` module, builds an internal import dependency graph
tracking only project-internal imports, and infers the tech stack from
dependency files. Produces a `CodebaseManifest` вЂ” a fully validated Pydantic
model saved to `output/manifest.json`.

**Agent pipeline** вЂ” a LangGraph `StateGraph` with `DocState` as shared state.
Planner seeds state, five writer agents produce sections via Jinja2-rendered
prompts, critic scores each 1-10 and flags weak ones for revision, revision
router loops writers back with critique context, assembler collects all sections.

**Renderer layer** вЂ” takes `final_docs` dict and produces three formats:
`documentation.md` (single assembled Markdown), `site/` (multi-page static
HTML), and `project_documentation.pdf` (PDF via WeasyPrint).

**CLI** вЂ” Typer app with `autodoc run --input --output --format` flags.
Rich progress bar shows live pipeline execution. Quality score table printed
after critic completes.

### Key design decisions

**Iterative refinement loop** вЂ” critic scores each section independently.
Only sections below `REVISION_THRESHOLD=7` are sent back. Writers receive
specific feedback via `{{ critique }}` Jinja2 variable. Capped at `MAX_REVISIONS=2`.

**JSON-only critic response** вЂ” critic instructed to return raw JSON.
`_parse_response()` strips markdown fences defensively and returns empty
dicts on malformed JSON вЂ” pipeline never crashes due to LLM formatting.

**Strategy pattern for LLM** вЂ” `BaseLLMClient` ABC, one factory, one env var.
`AUTODOC_USE_MOCK=false` + key в†’ real Claude. Zero code changes required.

**Jinja2 for both prompts and output** вЂ” same `render_prompt()` pattern used
for agent prompts (`autodoc/prompts/`) and output templates (`autodoc/templates/`).

## Authentication and security

### Overview
AutoDoc is a local CLI tool with no user-facing authentication. Security
is limited to API key management and safe handling of cloned repositories.

### Security measures

| Measure | Implementation |
|---------|---------------|
| API key never hardcoded | Pydantic BaseSettings reads from `.env` only |
| API key never logged | No log statement references the key field |
| `.env` gitignored | Root `.gitignore` excludes `.env` |
| Temp repos cleaned up | `try/finally` in `cli.py` |
| Shallow clone only | `depth=1` in `git.Repo.clone_from()` |
| Mock mode default | `AUTODOC_USE_MOCK=true` вЂ” key never needed in dev |
| StrictUndefined Jinja2 | Prompt variable errors caught at dev time |
| JSON parsing hardened | Critic strips fences, returns empty on malformed |
